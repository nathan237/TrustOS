    but s_get_value and s_bulk_get_value do NOT
    → Asymmetric locking: writer locks, reader doesn't

 6. ATTACK VECTORS (priority order)
═══════════════════════════════════════════════════════════════════════

 VECTOR 1: RACE s_set_value vs s_get_value [HIGH PRIORITY]
 ──────────────────────────────────────────────────────────
 Neither function has any locking. Race them:
   Thread A: s_set_value(surface, key, large_OSData)
   Thread B: s_get_value(surface, key) → reads during replacement
 
 The race window in s_set_value:
   1. Loads inner object at self+0x18
   2. Auth vtable (AUTDA)
   3. Calls vtable+0x128 (property container lookup)
   4. Calls vtable+0x118 (property setter)
   5. Calls vtable+0x20 (retain new value)
 
 Between steps 3-4 or 4-5, a concurrent s_get_value can read stale data.
 
 VECTOR 2: TYPE CONFUSION s_set_value_xml → s_get_value [HIGH PRIORITY]
 ──────────────────────────────────────────────────────────────────────
 s_set_value_xml dispatches through different vtable offsets than s_set_value.
 If the XML deserializer creates a different object type than what
 s_get_value expects (e.g., OSDictionary where OSData expected):
   - The getter reads the wrong object layout
   - Pointers/sizes are misinterpreted
   - Controlled kernel memory read via fake backing pointer
 
 VECTOR 3: UAF via s_decrement_use_count race [MEDIUM]
 ─────────────────────────────────────────────────────
 No locking on decrement. Race with get_value:
   Thread A: decrement_use_count → refcount hits 0 → free
   Thread B: get_value → reads freed memory → info leak or crash
 
 VECTOR 4: s_bulk_get_value iteration race [MEDIUM]
 ──────────────────────────────────────────────────
 s_bulk_get_value has NO locking but iterates over multiple properties.
 If another thread modifies the property dictionary during iteration:
   - Array count vs actual entries desync → OOB read
   - Iterator invalidation → use of freed entries
 
 VECTOR 5: s_create_surface initialization race [LOW]
 ───────────────────────────────────────────────────
 No locking during surface creation. If increment_use_count runs
 before initialization completes, uninitialized fields may be used.
