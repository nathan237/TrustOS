8. OVERFLOW PROTECTION ANALYSIS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Total MUL-family instructions in IOSurface kext: 98
  UMULH/SMULH overflow checks: 15

  Protection patterns found:
    Pattern A: UMULH xhi, src1, src2 → CBNZ xhi → MUL (128-bit check)
    Pattern B: UMULL x, w1, w2 → LSR #32 → CBNZ (64→32-bit check)
    Pattern C: BL max_check_compute → LSR #32 → CBNZ (wrapper check)
    Pattern D: UMULL x, w1, w2 → CMP x, max → B.HI (range check)
    Pattern E: MUL w, w1, w2 → CMP x64, w32 (cross-check)

  IOSurface_max_check flow:
    1. Initializes output with sentinel 0xAAAAAAAAAAAAAAAA
    2. Calls max_check_compute (property lookup, NOT multiplication)
    3. result = max_check_compute(self, label, &output)
    4. if (output >> 32) != 0 → overflow detected, log + return 0
    5. else → STR w (truncate to 32-bit), return 1

  max_check call sites (10 total) check SECONDARY parameters:
    • IOSurfaceSubsampling
    • IOSurfacePlaneExtendedPixelsOnLeft/Right/Bottom/Top
    • IOSurfaceBufferTileFormat
    (NOT width, height, bpe, or bpr!)

  The main create function (0xfffffff00a1cb59c, ~700 insns) uses
  compiler-generated __builtin_mul_overflow() extensively:
    UMULH → CBNZ for 64×64 products
    UMULL → LSR#32 → CBNZ for 32×32 products
    ADDS → B.HS for addition carry

  ASSESSMENT: iOS 26.3 IOSurface has ROBUST overflow protection.
  No obvious integer overflow vulnerability in the main size
  computation paths. All critical multiplications are guarded.

9. ATTACK SURFACE ASSESSMENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  STRONG defenses:
    ✓ PAC on all vtable entries (IA + per-slot diversity)
    ✓ PAC on dispatch table (IA/0x705d)
    ✓ PAC on authenticated pointers (+0x390, DA/0xda2b)
    ✓ Compiler-generated overflow checks (UMULH + UMULL)
    ✓ BRK traps on PAC verification failure
    ✓ CASA/CASL atomic locking on container operations

  POTENTIAL attack vectors (ranked by feasibility):

  1. PROPERTY TYPE CONFUSION (HIGH priority)
     s_set_value (sel 4) and s_set_value_xml (sel 10) manipulate
     an OSDictionary of surface properties. If a property key
     can be set with one type and retrieved as another, type
     confusion yields arbitrary r/w.
     → Requires fuzzing the property interface

  2. RACE CONDITIONS (HIGH priority)
     Surface lifecycle: create → lock → set → get → unlock → release
     The lock at +0xd8 and CASA/CASL atomics suggest awareness of
     concurrency. But complex state machines often have windows:
     → Race between s_set_value and s_release_surface
     → Race between s_lock and property access
     → TOCTOU in surface array lookup (0xfffffff00a1eda10)

  3. IOConnectTrap REDIRECTION (MEDIUM priority)
     getTargetAndTrapForIndex at 0xfffffff00ad0ac49
     393 LDR+BL gadgets in IOSurface kext
     1098 BLRAA indirect calls — massive surface for PAC oracle

  4. HEAP FENG SHUI via s_set_value (MEDIUM priority)
     Spray IOSurface property values to control kernel heap layout.
     Use specific allocation sizes to target slab allocators.
     Corrupt adjacent heap metadata or objects.

  5. DMA/IOMMU MAPPING (LOW priority, requires entitlements)
     IOSurface buffers can be DMA-mapped. If the IOMMU mapping
     can be manipulated, direct physical memory access is possible.
     The flag at +0x3c0 bit 0x1a enables DMA path.

  6. UNPROTECTED MADD (LOW priority)
     MADD x8, x9, x20, x8 at 0xfffffff00a1f0724 has no visible
     overflow check and stores result directly. However, this is
     in a logging/diagnostic path, not the critical allocation path.

10. BOOTROM DUMP STRATEGY (REVISED)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  TARGET: Physical 0x100000000 (BootROM, 64KB)

  Given iOS 26.3's robust overflow protection, the path to BootROM
  requires either:

  Path A: Property Type Confusion → kernel R/W → ml_phys_read
    1. Fuzz s_set_value/s_get_value/s_set_value_xml for type confusion
    2. Achieve arbitrary kernel read/write via corrupted property object
    3. Locate kernel_task port in kernel memory
    4. Call ml_phys_read(0x100000000, 0x10000) to dump BootROM

  Path B: Race Condition → UAF → kernel R/W → ml_phys_read
    1. Race s_release_surface against property operations
    2. Spray replacement object into freed IOSurface slot
    3. Confused virtual dispatch → controlled function call
    4. Build kernel R/W primitive
    5. Call ml_phys_read for BootROM

  Path C: Alternative Kernel Bug Class
    1. Use IOSurface analysis as heap grooming primitive
    2. Trigger vulnerability in different driver/subsystem
    3. Use arbitrary R/W to call ml_phys_read

  KEY PRIMITIVES AVAILABLE:
    • ml_phys_read:  0xfffffff00814f740 (113 insns, reads from phys)
    • ml_phys_write: 0xfffffff00814f9f0 (111 insns, writes to phys)
    • gPhysBase:     [0xfffffff007b00bb8] (physical base address)
    • gPhysEnd:      [0xfffffff007b00bc0] (physical end address)

  ml_phys_read bypasses IOMMU — it uses direct physical addressing
  via the kernel's physmap window. Once you have kernel code execution
  or arbitrary kernel R/W, calling ml_phys_read with address
  0x100000000 dumps the A13 BootROM.
